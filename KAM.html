KAM

Knowledge Application Methodology

A four-layer decoupled framework that transforms domain expertise into AI-ready capabilities — conquering the final barrier of the most difficult automation tasks.

v1.0 · Methodology Specification
⚡

In complex, long-chain AI automation, the real bottleneck is never code — it's knowledge. Raw domain expertise is fragmented, implicit, and locked inside experts' heads.

KAM introduces Knowledge Engineering as the missing discipline: strip an expert's "intuition," clean it, and reconstruct it into machine-consumable logical assets.

Knowledge = Σ { Specific Information + Data Flow + Control Flow }

Decouple what experts know from how engineers build.
Make AI development reusable, replaceable, and independently evolving.

Core Concepts

Three pillars that define the KAM framework

🎯
KAM
A four-layer decoupled methodology for transforming domain expertise into AI-ready capabilities. Separates general practices from domain-specific components.
📦
Knowledge
The sum of Specific Information + Data Flow + Control Flow. Decouples domain information from AI tool development. Can be internalized (training) or externalized (context/RAG).
🔬
Knowledge Engineering
The entire process of converting raw, fragmented, implicit domain data into standardized, AI-consumable "knowledge blocks." Pseudocode written by domain experts.

Knowledge Forms & Next Steps

How knowledge enters and evolves within the AI system

🧬
Internalized
Inserted into the model via training or fine-tuning. Deep effect, low flexibility.
💉
Externalized
Dynamically injected via context prompts, RAG, or agent skills at runtime.
📊
Datasets
Processed into training or fine-tuning datasets for model absorption.
🔄
Flows
Structured as processes and logic chains for deterministic execution.
🤖
Agents
Orchestrated into autonomous planning loops for open-ended tasks.
🔧
Skills / MCPs
Packaged as capability modules connecting professional and general tools.

The Four-Layer Pipeline

Decouple professional knowledge from engineering implementation

💻 L4 · Developer
L1
Professional Knowledge Layer
Domain Expert
Experts reorganize their knowledge for machine consumption based on real, cutting-edge workflows. They log work by the minute — detailing Time + Action + Purpose. No AI understanding required, but must provide initial data and label definitions.
← Own domain knowledge & experience → Cases, samples, logs, label definitions ✓ Data covers core scenarios, labels unambiguous
L2
Knowledge Processing Layer
Knowledge Engineer
Raw knowledge cannot automatically become usable knowledge. Build domain knowledge maps, uncover hidden premises, plan control flow & data flow, annotate, clean, and split into standardized Knowledge Blocks.
← Initial data & labels from L1 → Knowledge Blocks + flow plans ✓ Complete hierarchy, implicit knowledge explicit
L3
AI Tool Planning Layer
AI Architect
Assess feasibility & cost of technical routes. Forecast performance ceilings. Design evaluation metrics, test sets, and rollback plans. Choose between custom training, commercial APIs, RAG architecture, and prompt ratios.
← Knowledge blocks & flow plans from L2 → Solution doc: routes, budget, risk plans ✓ Immediately executable, clear eval & fallback
L4
Tool Development Layer
Development Engineer
General dev (100% AI): interaction design, system architecture, deployment. Dedicated dev (100% domain): professional software APIs, proprietary formats, domain-specific workflow orchestration.
← Solution document from L3 → Executable AI Tool in production ✓ Meets functional & performance metrics

Feedback & Traceback

When issues arise, trace back to the responsible layer

🔄 Layer-to-Layer Feedback Mechanism
Tool performance fails metrics
→ Trace to Layer 3
Re-evaluate the architectural plan and technical routes.
Plan cannot be designed
→ Trace to Layer 2
Reprocess knowledge, adjust granularity, or improve annotations.
Missing foundational concepts
→ Trace to Layer 1
Request experts to supplement data or source textbook knowledge.
All routes too costly / ineffective
→ External Dependency
Wait for tool/model evolution or adjust the project's target scope.

Data vs. Knowledge

A poetry example to illustrate the difference

📝 Appendix: Writing a Poem
Raw Data
  • A collection of poems
  • Classic literature anthologies
  • Dictionaries & references
  • Specific excerpts from classics
  • Word-by-word annotations
KAM Knowledge (Hidden Premises)
  • What actually defines a poem (e.g., rhyming)
  • How to judge if a poem is good (emotion, imagery)
  • The exact step-by-step process of writing
  • Boundary: not an essay, limerick, or paper
  • Tool awareness: pen vs. computer
  • Operational flow: boot → open editor → type → line breaks → Enter → translate
